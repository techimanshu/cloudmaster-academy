Exam focus: Transit Gateway vs VPC Peering vs PrivateLink is heavily tested. Know exactly when each applies and their limitations.
Core networking services
VPC fundamentals
  • CIDR cannot overlap for peering
  • VPC peering: non-transitive — A↔B, B↔C ≠ A↔C
  • Transit Gateway solves transitivity at scale
  • Subnet = AZ-bound; one subnet per AZ minimum
  • NAT Gateway: per-AZ, place one per AZ for HA
  • Security Groups: stateful · NACLs: stateless
Transit Gateway
  • Regional hub for VPCs, VPNs, Direct Connect
  • Supports inter-region peering
  • Route tables control which attachments communicate
  • Multicast support (unique to TGW)
  • Use when you have 5+ VPCs to connect
  • Replaces complex full-mesh VPC peering
Direct Connect
  • 1 Gbps or 10 Gbps dedicated connection
  • Hosted connection: sub-1G via partner
  • VIFs: Private (VPC) · Public (AWS services) · Transit (TGW)
  • For HA: two DX connections OR DX + VPN backup
  • Direct Connect Gateway spans multiple regions
  • DX alone is NOT redundant by default
Route 53
  • Simple: single resource, no health checks
  • Weighted: % traffic split, A/B testing
  • Latency: lowest latency to user
  • Failover: primary/secondary, requires health check
  • Geolocation: based on user location
  • Geoproximity: location + bias adjustment
  • Alias records: free, for AWS resources (ELB, CF, S3)
Load balancers
  • ALB: HTTP/HTTPS, path/host routing, WAF integration
  • NLB: TCP/UDP, ultra-low latency, static IP
  • GLB: deploy 3rd-party appliances (IDS/IPS)
  • NLB preserves source IP; ALB does not by default
  • Cross-zone LB: on by default ALB, off for NLB
  • NLB supports UDP and static Elastic IPs
PrivateLink
  • Expose service to other VPCs without full VPC access
  • No VPC peering, no IGW, no NAT needed
  • Provider creates endpoint service; consumer creates interface endpoint
  • Supports cross-account and cross-region (via TGW)
  • Use when sharing SaaS-style services across accounts
CloudFront
  • Origin failover: primary + secondary origin group
  • Lambda@Edge: 13 locations, full Node.js/Python
  • CF Functions: lightweight JS, 47+ edge locations
  • Signed URLs: one object, expiry, IP restriction
  • Signed Cookies: multiple objects, same domain
  • OAC replaces OAI for S3 origin access control
Global Accelerator
  • 2 static anycast IPs, routes via AWS backbone
  • Works with TCP/UDP (not just HTTP)
  • Instant regional failover (<30 seconds)
  • vs CloudFront: GA for non-HTTP or IP whitelisting
  • Health checks built in, automatic failover

DNS Resolver — Key concepts
Inbound Endpoint
  • Where: ENI created IN your VPC (private subnet)
  • Who uses it: On-prem DNS servers point conditional forwarder at its IP
  • One endpoint = multiple ENIs (one per AZ) for HA
  • On-prem must have DX or VPN to reach ENI private IPs
  • Resolves: Private Hosted Zones, VPC-local DNS (ip-x.region.compute.internal)
  • Exam trap: Do NOT put inbound endpoint on-prem — it lives in AWS VPC
Outbound Endpoint
  • Where: ENI created IN your VPC (private subnet)
  • Who uses it: Route 53 Resolver itself — when a forwarding rule matches
  • Must pair with a Forwarding Rule — endpoint alone does nothing
  • Forwarding rule specifies: domain suffix + target on-prem DNS IPs
  • Resolves: on-prem domains e.g. corp.internal, ad.company.com
  • Exam trap: Outbound endpoint is still in AWS VPC, not on-prem
Resolver Rules sharing
  • Resolver rules can be shared via AWS RAM across accounts
  • Share a single FORWARD rule to all VPCs in your org
  • Centralised endpoint in shared-services VPC — all spoke VPCs use it via TGW
  • Rule priority: most specific domain wins (corp.internal beats .internal)
  • Rules are associated with VPCs — must associate per VPC to take effect
Private Hosted Zone (PHZ)
  • PHZ associated with one or more VPCs for private DNS resolution
  • Can associate PHZ with VPCs in different accounts using CLI/API
  • Split-horizon DNS: same domain, different records for public vs VPC
  • VPC must have enableDnsHostnames + enableDnsSupport = true
  • PHZ + Inbound Endpoint = on-prem resolves to private AWS IPs
VPC DNS settings
  • enableDnsSupport — enables Route 53 Resolver (169.254.169.253 or VPC+2)
  • enableDnsHostnames — assigns DNS names to EC2 public IPs
  • Both must be true to use Private Hosted Zones in a VPC
  • VPC+2 address = built-in DNS resolver (e.g. 10.0.0.2 for 10.0.0.0/16)
  • Custom DHCP option set can override DNS server — use with caution
Common hybrid DNS patterns
  • On-prem resolves AWS: Inbound endpoint + PHZ + conditional forwarder on-prem
  • AWS resolves on-prem: Outbound endpoint + Forwarding Rule pointing to on-prem DNS
  • Both directions: Need BOTH inbound AND outbound endpoints (different endpoints)
  • Centralised DNS: One shared VPC holds both endpoints; shared via RAM + TGW routing
  • DNS over Direct Connect: Private VIF routes DNS traffic — no internet required
Exam scenario — DNS resolver traps
On-prem servers cannot resolve db.app.internal which is in a Route 53 Private Hosted Zone. They connect to AWS via Direct Connect. What do you need?
Route 53 Resolver Inbound Endpoint + on-prem conditional forwarder — Create an Inbound Endpoint (ENI in your VPC). Configure on-prem DNS server with a conditional forwarder: domain app.internal → Inbound Endpoint IP. Ensure DX private VIF allows DNS traffic on port 53. The PHZ must be associated with the VPC containing the endpoint.
EC2 instances in a VPC cannot resolve ldap.corp.internal — a domain managed by on-prem Active Directory. What is the fix?
Route 53 Resolver Outbound Endpoint + Forwarding Rule — Create an Outbound Endpoint (ENI in your VPC). Create a Resolver Forwarding Rule: corp.internal → 10.0.0.53 (on-prem AD DNS IP). Associate the rule with the VPC. Ensure VPN or DX allows UDP/TCP 53 from the outbound endpoint ENI IPs to the on-prem DNS.
You have 20 VPCs across 5 accounts. All need to resolve on-prem domains. Setting up outbound endpoints in every VPC is too expensive. How do you centralise?
Centralised outbound endpoint in shared-services VPC + RAM-shared Forwarding Rules + Transit Gateway — Create one Outbound Endpoint in a shared-services VPC attached to TGW. Share the Forwarding Rule via AWS RAM to all 20 VPCs. Route DNS traffic from spoke VPCs to shared VPC via TGW route table. One endpoint serves all accounts — no per-VPC endpoints needed.

Compute services
EC2 purchasing
  • On-demand: flexible, no commitment, highest cost
  • Reserved: 1-3yr, up to 72% off, Standard > Convertible discount
  • Spot: up to 90% off, 2-min interruption notice
  • Dedicated Host: compliance, per-socket/core licensing
  • Dedicated Instance: hardware not shared, no visibility
  • Spot Fleets + Capacity Rebalancing for resilient spot
Auto Scaling
  • Target tracking: maintain a metric value (simplest)
  • Step scaling: tiered response to alarm breaches
  • Scheduled: known traffic patterns, predictable
  • Predictive: ML-based, scale before demand hits
  • Lifecycle hooks: pause during launch or terminate
  • Warm pools: pre-initialized instances reduce latency
Lambda
  • Max 15 min execution, 10 GB memory, 512 MB–10 GB /tmp
  • Reserved concurrency: cap function, protect downstream
  • Provisioned concurrency: pre-warm for consistent latency
  • VPC Lambda: needs ENI, adds cold-start — use provisioned
  • Destinations: route on success/failure to SQS/SNS/EventBridge
  • Layers: shared dependencies up to 5 per function
ECS vs EKS
  • ECS: simpler, AWS-native, Task Definitions
  • EKS: Kubernetes, portability, more control
  • Fargate: serverless containers, no EC2 management
  • ECS on EC2: spot support, GPU workloads, more control
  • App Mesh: service mesh for both ECS and EKS
  • ECR: managed container registry, integrated with both
Elastic Beanstalk
  • PaaS — manage the app, not the infrastructure
  • All-at-once: fastest, downtime during deploy
  • Rolling: batches, reduced capacity during deploy
  • Rolling + batch: maintains capacity throughout
  • Immutable: safest, new instances first, then swap
  • Blue/Green: swap CNAME, full environment swap
Edge & hybrid compute
  • Outposts: AWS rack on-premises, same APIs as AWS
  • Wavelength: embed compute at 5G carrier edge
  • Local Zones: AWS infra in metro areas for latency
  • Use Outposts for data residency or ultra-low on-prem latency
  • Local Zones: media & entertainment, real-time gaming

Storage services
S3 storage classes
  • Standard: frequent access, 3 AZ, 99.99% availability
  • Standard-IA: infrequent, 3 AZ, retrieval fee applies
  • One Zone-IA: infrequent, 1 AZ, 20% cheaper than IA
  • Glacier Instant: archives with millisecond retrieval
  • Glacier Flexible: 1–12hr retrieval, cheapest archive
  • Glacier Deep Archive: 12–48hr, absolute lowest cost
  • Intelligent-Tiering: auto-moves between tiers, no retrieval fee
S3 features
  • Versioning: required for MFA delete and replication
  • SRR: same-region replication · CRR: cross-region
  • Object Lock — Governance: can be bypassed by privileged users
  • Object Lock — Compliance: cannot be overridden, even by root
  • Transfer Acceleration: CloudFront edge for uploads
  • Multipart upload: required >5GB, recommended >100MB
  • S3 Select / Glacier Select: query in-place with SQL
EBS volume types
  • gp3: default SSD, independent IOPS + throughput tuning
  • gp2: legacy, IOPS tied to size (3 IOPS/GB)
  • io2 Block Express: max 256K IOPS, Multi-Attach capable
  • st1: throughput HDD, sequential reads (data warehouse)
  • sc1: cold HDD, lowest cost, infrequent access
  • Snapshots: incremental, stored in S3, copy cross-region
EFS vs FSx family
  • EFS: NFS, Linux only, multi-AZ, auto-scales
  • FSx for Windows: SMB, AD integration, Windows workloads
  • FSx for Lustre: HPC, ML training, integrates with S3
  • FSx for ONTAP: NFS+SMB+iSCSI, multi-protocol, snapshots
  • FSx for OpenZFS: NFS, Linux, ZFS features
  • FSx for Lustre: sub-millisecond latency for HPC
Snow family
  • Snowcone: 8 TB, smallest, DataSync agent included
  • Snowball Edge Storage: 80 TB, bulk transfer
  • Snowball Edge Compute: EC2 + Lambda at edge
  • Snowmobile: exabyte-scale, physical truck, 100 PB
  • Rule: use Snow when transfer time >1 week over network
Storage Gateway
  • S3 File Gateway: NFS/SMB files → S3 objects
  • FSx File Gateway: local cache for FSx Windows
  • Volume (Cached): primary in S3, cache subset on-prem
  • Volume (Stored): primary on-prem, async S3 backup
  • Tape Gateway: virtual tape library → S3/Glacier
  • DataSync for migration; Gateway for ongoing hybrid access

Database services
RDS
  • Multi-AZ: synchronous standby, automatic failover only
  • Read Replicas: async, for read scaling — NOT high availability
  • Multi-AZ + Read Replica = HA + read scaling combined
  • Automated backups: 0–35 days retention window
  • Cannot SSH into RDS instances — fully managed
  • Multi-AZ failover: typically 1–2 minutes
Aurora
  • 6 copies across 3 AZs, auto-heals storage
  • Up to 15 read replicas with auto-scaling Reader endpoint
  • Aurora Serverless v2: auto-scales capacity in ACUs
  • Global Database: 1 primary + up to 5 secondary regions
  • Global Database replication lag: <1 second typically
  • Backtrack: rewind DB in-place — no restore needed
DynamoDB
  • On-demand: unpredictable traffic, pay per request
  • Provisioned + Auto Scaling: predictable, cheaper
  • Eventually consistent reads (default) vs Strongly consistent
  • Global Tables: multi-region active/active replication
  • DynamoDB Streams: change capture → Lambda triggers
  • DAX: in-memory cache, microsecond read latency
ElastiCache
  • Redis: persistence, replication, Sorted Sets, pub/sub, Lua
  • Memcached: pure cache, horizontal scaling, multi-thread
  • Redis Cluster mode: sharding for massive datasets
  • Redis Global Datastore: cross-region replication
  • Use for: session store, leaderboards, rate limiting, DB offload
Analytics databases
  • Redshift: columnar, OLAP, petabyte-scale data warehouse
  • Redshift Spectrum: query S3 directly from Redshift
  • Athena: serverless SQL on S3, pay per query scanned
  • EMR: managed Hadoop/Spark for big data processing
  • OpenSearch: search + log analytics (replaces Elasticsearch)
Specialty databases
  • DocumentDB: MongoDB-compatible, managed, JSON
  • Keyspaces: Cassandra-compatible, serverless
  • Neptune: graph database, social networks, fraud detection
  • QLDB: immutable ledger, cryptographic verification
  • Timestream: time-series, IoT telemetry, auto-tiering

Security & identity
IAM policy evaluation order
  • 1. Explicit Deny — always wins, period
  • 2. SCP (Organizations) — must allow action
  • 3. Resource-based policy — S3 bucket, SQS, etc.
  • 4. Identity-based policy — attached to user/role
  • 5. Permissions boundary — caps what identity can do
  • 6. Session policy — assumed role session scope
  • Default result: implicit deny everything
Cross-account access
  • IAM Role + AssumeRole = preferred and recommended method
  • Resource-based policy works for S3, SQS, Lambda
  • Role trust policy must allow source account/principal
  • External ID: prevents confused deputy attack
  • SCPs apply to member accounts — NOT management account
  • Delegated admin: member account manages specific service
KMS
  • CMK (Customer Managed): full control, rotation, key policy
  • AWS Managed: auto-rotated annually, per-service keys
  • AWS Owned: fully managed by AWS, no customer visibility
  • Multi-Region keys: same key material, different regions
  • Envelope encryption: CMK encrypts data key → data key encrypts data
  • CloudHSM: dedicated HSM hardware, FIPS 140-2 Level 3
Threat detection
  • GuardDuty: ML threat detection, CloudTrail + VPC Flow + DNS
  • Inspector: vulnerability scanning EC2, ECR images, Lambda
  • Macie: S3 sensitive data discovery — PII, financial data
  • Security Hub: aggregated findings across accounts and regions
  • Detective: investigate and visualize security findings
  • Enable GuardDuty org-wide from Organizations delegated admin
Network security
  • Security Groups: stateful, instance-level, allow rules only
  • NACLs: stateless, subnet-level, allow + explicit deny
  • WAF: Layer 7 rules, attach to ALB/CloudFront/API Gateway
  • Shield Standard: free DDoS protection, always on
  • Shield Advanced: $3K/mo, 24/7 DRT team, cost protection
  • Firewall Manager: central WAF + Shield policy management
Secrets & certificates
  • Secrets Manager: auto-rotation, cross-account, RDS native integration
  • Parameter Store: cheaper, no auto-rotation, hierarchical paths
  • ACM: free SSL/TLS certs for ALB/CloudFront — cannot export private key
  • CloudHSM: dedicated hardware, you control key material
  • Use Secrets Manager when rotation is required by compliance

High availability & disaster recovery
DR strategy tiers
  • Backup & Restore: RPO hours · RTO hours · lowest cost
  • Pilot Light: RPO minutes · RTO tens of minutes
  • Warm Standby: RPO seconds · RTO minutes · scaled-down copy
  • Multi-site Active/Active: RPO ~0 · RTO ~0 · highest cost
  • Justify choice against RTO/RPO numbers given in question
  • Cost increases with each tier up the stack
Aurora HA
  • 6 copies of data across 3 AZs automatically
  • Failover: <30 seconds — promotes read replica
  • Global Database: RPO <1s, RTO <1min for cross-region
  • Cluster volume heals itself — no storage disruption
  • Backtrack: rewind in-place without snapshot restore
Route 53 failover
  • Active-passive: primary + secondary, health check on primary
  • Active-active: weighted or latency, all endpoints healthy
  • Health check types: endpoint · CloudWatch alarm · calculated
  • Lower TTL before planned failover event window
  • String matching: check response body for specific content
Backup & replication
  • AWS Backup: centralized, policy-based, cross-account/region
  • EBS snapshots: incremental, copy cross-region for DR
  • RDS: point-in-time restore within retention window (0–35 days)
  • S3 CRR: async replication to another region bucket
  • DynamoDB: PITR up to 35 days, on-demand backup
SLA quick reference
  • S3 Standard: 11 nines durability (99.999999999%)
  • S3 Standard: 99.99% availability SLA
  • EC2: 99.99% uptime per region
  • RDS Multi-AZ failover: 1–2 minutes typical
  • Aurora failover: <30 seconds to read replica
  • DynamoDB: 99.999% availability (Global Tables)
Global Accelerator vs CloudFront
  • Both use AWS global network and edge locations
  • CloudFront: caching, HTTP/S, static + dynamic content
  • GA: no caching, TCP/UDP, non-HTTP workloads
  • GA: static anycast IPs — useful for IP whitelisting
  • GA: instant failover <30s · CF: DNS-based, slower

Migration services & strategies
The 7 Rs
  • Retire: decommission — no longer needed
  • Retain: keep on-prem (compliance, not ready)
  • Relocate: move as-is to VMware Cloud on AWS
  • Rehost (lift & shift): move to EC2 — quickest, least change
  • Replatform: minor optimizations (RDS vs self-managed DB)
  • Repurchase: move to SaaS (Salesforce, Workday, ServiceNow)
  • Refactor / Re-architect: rebuild cloud-native, highest ROI
MGN — Application Migration
  • Lift-and-shift physical/virtual/cloud servers to AWS EC2
  • Continuous block-level replication while source stays live
  • Cutover with minimal downtime window
  • Replaces older Server Migration Service (SMS)
  • Use for: Rehost strategy at scale
  • Works with Windows and Linux servers
DMS — Database Migration
  • Source database stays live during migration
  • Homogeneous: same engine (MySQL → RDS MySQL)
  • Heterogeneous: different engine — use SCT first for schema
  • Schema Conversion Tool (SCT) converts schema + stored procs
  • CDC (Change Data Capture): ongoing replication after migration
  • Supports S3, Redshift as targets for data lake migration
Migration Hub
  • Central tracking dashboard for all migration workstreams
  • Works with MGN, DMS, and certified partner tools
  • Migration Hub Orchestrator: workflow automation for migrations
  • App Discovery Service: inventory on-prem servers and dependencies
  • Agentless (VMware vCenter) vs Agent-based discovery modes
DataSync
  • Online file/object transfer to AWS — scheduled or one-time
  • Sources: NFS, SMB, HDFS, S3-compatible, self-managed
  • Targets: S3, EFS, FSx for Windows/Lustre/ONTAP/OpenZFS
  • Preserves metadata and permissions during transfer
  • vs Storage Gateway: DataSync for migration; Gateway for ongoing hybrid
VMware Cloud on AWS
  • Run VMware SDDC on AWS bare metal hardware
  • No re-platforming — same vSphere, NSX, vSAN tooling
  • Low latency to AWS native services via ENI connection
  • Use case: data center extension, evacuation, VMware DR
  • HCX for live migration of VMs from on-prem

AWS Organizations — Structure & Concepts
Organization hierarchy
  • Root: top-level container — one per org, cannot be deleted
  • Management (Master) account: owns the org, pays consolidated bill
  • OUs (Organizational Units): containers for accounts or nested OUs
  • Member accounts: leaf nodes — can only be in one OU at a time
  • Typical OU structure: Root → Security / Workloads / Sandbox / Suspended
  • Workloads OU → Prod / Non-Prod / Dev → individual account per team/service
  • Max OU nesting depth: 5 levels below Root
Management account rules
  • SCPs NEVER apply to the management account — even if attached to Root
  • Management account cannot be moved to an OU
  • Use management account ONLY for billing and org management — no workloads
  • Delegated administrator: assign a member account to manage a service instead
  • Services with delegated admin support: GuardDuty, Security Hub, Config, Macie, RAM, IAM Identity Center
Consolidated billing
  • All accounts' bills roll up to management account
  • RI/SP sharing: unused reservations automatically shared org-wide
  • Turn off RI sharing per account if needed (billing console)
  • Volume discounts: combined usage across all accounts reaches tier thresholds
  • S3, data transfer, EC2 usage pooled for discount calculation
  • Management account sees full cost breakdown by linked account
AWS RAM with Organizations
  • Share resources without resource duplication across accounts
  • Shareable resources: Subnets, TGW, Route 53 Resolver Rules, License configs, etc.
  • Share to entire org, an OU, or specific account
  • Enable resource sharing in Organizations console first
  • Exam pattern: Shared VPC subnets via RAM — multiple accounts deploy into same VPC but each controls their own resources
Control Tower
  • Opinionated, automated setup of a multi-account AWS environment
  • Builds a Landing Zone: baseline OU structure + guardrails + logging accounts

Service Control Policies (SCPs) — Deep Dive
📐 SCP Inheritance Model

Effective permissions = IAM policy ∩ ALL SCPs in the hierarchy (root → OU → account). Each level must ALLOW for an action to be permitted.

LevelSCP attached here affects…Note
RootALL accounts in org (except management)Broadest scope — use for org-wide guardrails
OUAll accounts in that OU and nested OUsInherits Root SCPs + adds its own
AccountOnly that specific member accountInherits all parent SCPs
Management accountNEVER affected by any SCPEven if SCP attached to Root
SCP fundamentals
  • SCPs are maximum permission boundaries — they do NOT grant permissions
  • An Allow in SCP doesn't give access — you still need an IAM Allow
  • A Deny in SCP blocks even root user in member accounts
  • SCPs apply to all principals in the account: IAM users, roles, root user
  • SCPs do NOT apply to service-linked roles
  • SCPs affect actions, not resource-based policies (S3 bucket policy still needs IAM policy)
Deny-list strategy (default)
  • Default: FullAWSAccess SCP attached to Root — allows everything
  • Add specific Deny SCPs at OU/account level to restrict
  • Simpler to manage: only write policies for what you want to block
  • Example: Deny ec2:* in Sandbox OU to prevent production-like workloads
  • Example: Deny cloudtrail:StopLogging org-wide so no one can disable audit trail
  • Most common approach for most organizations
Allow-list strategy
  • Remove FullAWSAccess from Root — everything denied by default
  • Attach explicit Allow SCPs listing permitted services at each OU/account
  • More restrictive: if service not in SCP allow list, it's blocked regardless of IAM
  • Example: Prod OU SCP allows only ec2:*, rds:*, s3:*, cloudwatch:*
  • Complex to maintain — easy to accidentally block needed services
  • Used in highly regulated environments (PCI, HIPAA) where you want whitelist-only access
SCP condition keys
  • aws:RequestedRegion — restrict actions to specific regions
  • aws:PrincipalArn — allow only specific roles to perform action
  • aws:PrincipalTag — condition on tag attached to the principal
  • ec2:Region — EC2-specific region restriction
  • Example: Deny all except us-east-1 and us-west-2 for all member accounts
  • Not — use StringNotEquals with aws:RequestedRegion to deny all other regions
SCP vs IAM vs RCP
  • SCP: restricts what principals IN an account can do — affects identity
  • IAM Policy: grants permissions to specific users/roles
  • Resource Control Policy (RCP): NEW — restricts access TO resources org-wide (like SCP but for resources)
  • Permission boundary: IAM-only, per entity — max limit for a single role/user
  • SCP + IAM = AND logic — both must allow
  • Resource-based policies (S3, KMS) can bypass SCP if cross-account — SCP applies to the requester's account
SCP cannot block
  • Actions by management account — SCPs don't apply
  • Actions by service-linked roles — e.g. AWSServiceRoleForElasticLoadBalancing
  • Actions on resources in OTHER accounts (cross-account) — SCP governs the requester only
  • IAM Identity Center permission sets — evaluated separately
  • Root user of management account — fully unrestricted

Organizations policy types
Policy type overview
  • SCPs: max permission boundary for IAM principals in accounts
  • Tag policies: enforce tag key naming and allowed values org-wide
  • Backup policies: define AWS Backup plans and apply across accounts
  • AI opt-out policies: control whether AI services can use data for training
  • Resource Control Policies (RCP): restrict access to resources regardless of IAM — new, powerful
  • Each policy type must be enabled in org features before attaching
Tag policies
  • Define required tag keys (e.g. Environment, CostCenter)
  • Define allowed values (e.g. Prod, Dev, Staging)
  • Enforce case sensitivity of tag keys
  • Works with: EC2, S3, RDS, Lambda, and most taggable resources
  • Does NOT auto-tag — it validates tags on resource operations
  • Non-compliant resources shown in AWS Config Tag compliance view
Delegated Administrator
  • Management account delegates org-level API access to a member account
  • Allows member account to manage a service org-wide (e.g. Security Hub)
  • Supported services: GuardDuty, Security Hub, Config, Macie, RAM, Inspector, Firewall Manager
  • Best practice: use a dedicated security/audit account as delegated admin
  • Management account retains full control — delegation does not remove its access
Organizations scenario traps
A company attaches an SCP that denies all EC2 actions to the Root OU. The management account's admin still needs to launch EC2 instances. Will the SCP block them?
No — SCPs never apply to the management account — Even an SCP attached at the Root level does not restrict the management account. The admin can still launch EC2. This is a common exam trap. The fix if you want restrictions on management account is to move workloads OUT of management account — not use SCPs.
An SCP attached to the Prod OU has an explicit Allow for s3:GetObject. A member account IAM policy has Deny s3:GetObject. Can the user access S3?
No — explicit IAM Deny overrides SCP Allow — SCPs set the maximum boundary but IAM evaluation still applies within that boundary. An explicit Deny anywhere in the IAM policy chain (even permission boundaries) always wins. The SCP Allow means the action is not blocked at org level, but the IAM Deny within the account still prevents it.
You need to ensure no account in your organization can disable AWS Config recording. Which is the minimum viable solution?
SCP at Root level denying config:StopConfigurationRecorder and config:DeleteConfigurationRecorder — Attach the SCP to Root so it applies to all member accounts. SCPs override even admin IAM users in those accounts. Remember this SCP will NOT apply to the management account itself — run Config separately there or use Control Tower mandatory guardrail which enforces it.
Strategy: SAP questions are scenario-based. Eliminate obviously wrong answers first, then choose the most operationally efficient, scalable, and cost-effective option.
Common scenario traps
Need to connect 50 VPCs across 3 accounts. What is best?
Transit Gateway — VPC peering is non-transitive and becomes unmanageable at scale (50 VPCs = 1225 peering connections). TGW centralizes routing with per-OU route table isolation.
Application needs consistent low-latency connectivity to on-prem. Internet VPN is unstable.
Direct Connect — dedicated private connection, consistent bandwidth. Add a VPN as backup for HA. DX alone is NOT redundant — need two DX connections for full HA.
10 TB of data needs to move to S3. Transfer would take 3 weeks over existing bandwidth.
Snowball Edge — rule of thumb: if transfer time exceeds 1 week, Snow family is faster and cheaper. 3 weeks >> threshold. Snowball physically ships data to AWS.
Multi-region app — users must hit nearest region. Need fast failover if a region fails.
Route 53 Latency routing + Health Checks or Global Accelerator. GA preferred for non-HTTP, static IPs, or sub-30s failover. R53 for DNS-level geo control and caching.
Prevent member accounts from disabling CloudTrail, even for admin users.
SCP (Service Control Policy) in AWS Organizations — SCPs override IAM permissions in all member accounts. IAM alone cannot enforce org-wide restrictions. SCPs are the only guardrail that works here.
Share a microservice with 100 VPCs without exposing network access.
PrivateLink (VPC Endpoint Service) — exposes only the specific service endpoint. No CIDR overlap issues, no full VPC peering, works cross-account. Scales to thousands of consumers.
RDS failover is taking 2 minutes. Need sub-30-second failover for MySQL.
Migrate to Aurora MySQL — Aurora failover is <30 seconds by promoting a read replica. RDS Multi-AZ typical failover is 1–2 minutes. Aurora also offers up to 15 replicas.
Compliance requires all S3 objects to be immutable for 7 years. No exceptions.
S3 Object Lock in Compliance mode — Compliance mode cannot be overridden even by root user. Governance mode can be bypassed by privileged IAM users with s3:BypassGovernanceRetention.
Migrate Oracle DB to AWS with minimum application code changes, reduce licensing cost.
RDS for PostgreSQL + SCT + DMS — SCT converts Oracle schema/stored procs to PostgreSQL. DMS migrates data with CDC for live cutover. Eliminates Oracle licensing cost. Replatform strategy.
Lowest possible RTO/RPO across regions. Cost is not the primary constraint.
Multi-site active/active — both regions serve traffic simultaneously. Combine Aurora Global Database + Route 53 active-active + Global Accelerator. Most expensive but near-zero RTO/RPO.
Serverless app needs to process SQS messages but also handle failures gracefully.
Lambda with SQS trigger + Dead Letter Queue (DLQ) — configure DLQ on SQS or Lambda Destinations for failed invocations. Set maxReceiveCount before DLQ routing. Use Lambda reserved concurrency to protect downstream services.
Windows file server workload on-prem. Need to migrate to AWS with AD integration.
FSx for Windows File Server — native SMB, Active Directory integration, DFS namespaces. Use AWS Managed Microsoft AD or trust with on-prem AD. Do not use EFS — Linux only.
Real-world scenario bank — new questions
A financial services company runs batch jobs every night that process 10 TB of data. Jobs take 6 hours. The team wants to cut compute cost by 70% without changing the code.
EC2 Spot Instances via AWS Batch — Spot gives up to 90% savings. AWS Batch manages spot interruptions by checkpointing and retrying. For 6-hour nightly batch workloads with no SLA, this is the canonical cost-cut answer. Avoid on-demand or Reserved here — you're not running 24/7.
A startup has a monolithic app on EC2. They want to migrate to containers without refactoring, keep ops overhead minimal, and avoid managing EC2 instances.
ECS on Fargate — lift-and-shift into containers without managing underlying EC2 hosts. No cluster EC2 management, no patching. EKS Fargate is an option but adds Kubernetes complexity unnecessarily for a migration scenario. Fargate = serverless containers.
An e-commerce company wants to ensure their checkout API is never throttled during Black Friday. Lambda concurrency is hitting limits and causing 429 errors.
Reserved Concurrency + Provisioned Concurrency — Reserved concurrency guarantees a fixed pool exclusively for checkout, preventing starvation. Provisioned concurrency pre-warms Lambda to eliminate cold starts under sudden load. Do both: reserve the quota, then provision within it for latency.
A company has 3 AWS accounts (Dev, Staging, Prod). Developers in Dev need to assume a role in Prod to read CloudWatch logs — without creating IAM users in Prod.
Cross-account IAM Role with sts:AssumeRole — Create a role in Prod with a trust policy allowing the Dev account principal. Developers in Dev assume this role via STS. No IAM users needed in Prod. This is the AWS-recommended cross-account access pattern for the exam.
A healthcare company stores patient data in S3. Regulation requires all data at rest and in transit to be encrypted. Encryption keys must be stored in a FIPS 140-2 Level 3 validated HSM.
AWS CloudHSM + SSE-C (or custom key store via KMS) — CloudHSM is FIPS 140-2 Level 3. Use CloudHSM-backed custom key store in AWS KMS, then apply SSE-KMS on S3. Standard KMS HSMs are Level 2. For strict Level 3 compliance, CloudHSM is the only option.
A SaaS product serves 500 enterprise customers. Each customer must have complete data isolation — no shared infrastructure at the database layer. But provisioning must be automated.
RDS per-tenant with AWS Service Catalog + CloudFormation StackSets — Service Catalog provides approved, pre-built "products" (CloudFormation templates). Provisioning a new customer triggers the stack that creates their dedicated RDS instance. Alternatively, Aurora serverless per tenant reduces cost while maintaining isolation.
An application stores session data in ElastiCache Redis. A region failure would cause all users to be logged out. How do you make session data survive a regional failure?
ElastiCache Global Datastore (Redis) — replicates Redis data across regions with <1 second latency. On regional failure, promote secondary cluster in DR region. Alternatively, move sessions to DynamoDB Global Tables with TTL — more resilient but higher latency for session reads.
A company wants to block specific countries from accessing their public API Gateway endpoint. No traffic from those regions should reach the backend at all.
AWS WAF Geo Match rule attached to API Gateway — WAF Geo Match blocks or allows requests based on country of origin at the edge. Attach WAF WebACL to API Gateway stage. CloudFront + WAF is also valid and blocks earlier. Do NOT use NACLs — they work on IP ranges, not countries.
An app on EC2 needs to call S3 and DynamoDB. A security audit flags that access keys are hardcoded in the application config files on the instance.
IAM Instance Profile (EC2 IAM Role) — attach an IAM role to the EC2 instance via an instance profile. The AWS SDK automatically retrieves temporary credentials from instance metadata (IMDS). Remove all hardcoded keys. This is the #1 security hygiene answer for EC2 → AWS service access.
A telecom company needs to process streaming clickstream data at 2 million events/sec, transform it in real time, and load it into Redshift and S3 simultaneously.
Kinesis Data Streams → Kinesis Data Firehose → S3 + Redshift — KDS ingests at scale with custom retention. Firehose delivers to S3 (raw) and Redshift (transformed). For real-time transformation within the pipeline, add Kinesis Data Analytics (Apache Flink) between streams and Firehose.
A DevOps team wants every EC2 instance launched in any account of their AWS Organization to automatically install the CloudWatch agent. No manual step per account.
AWS Systems Manager State Manager + EC2 Launch association, delegated via Organizations — State Manager associations can target all accounts/OUs via Systems Manager delegated admin. On instance launch, SSM applies the association to install the CW agent. Combine with EventBridge rule on EC2 state change for near-instant triggering.
A company's application writes 50,000 DynamoDB items/sec in bursts but reads are low. They are getting ProvisionedThroughputExceeded errors on writes.
Switch to DynamoDB On-Demand mode OR add SQS as a buffer — On-Demand auto-scales instantly for unpredictable spikes with no capacity planning. If cost is a concern for high sustained write rates, keep Provisioned + add SQS queue to smooth write bursts. Avoid hot partition keys — ensure even distribution via composite key design.
A media company transcodes videos uploaded to S3. Transcoding takes 2–4 hours per job. Sometimes multiple jobs pile up and servers sit idle between runs. How do you right-size the compute?
AWS Batch with Spot Instances + S3 event trigger via EventBridge — Batch dynamically provisions compute for each job and terminates it after. S3 PUT triggers EventBridge → Batch job submission. Long-running Spot jobs work well with Batch's retry/checkpointing. No idle servers between jobs — true scale-to-zero.
A company uses multiple AWS accounts. The security team wants a single pane of glass for all GuardDuty findings, Config non-compliance alerts, and Security Hub findings — aggregated in one account.
Security Hub with Organizations integration + designated administrator account — Enable Security Hub in all member accounts via Organizations, aggregate into a delegated admin account. GuardDuty and Config also support Organizations-level aggregation. EventBridge rules in the admin account can then route findings to Slack, PagerDuty, or SNS.
An on-prem application can only connect to resources via an IP address — no DNS resolution. It needs to access an internal ALB on AWS over Direct Connect. The ALB IP changes on scaling.
NLB in front of ALB (NLB → ALB as target), exposing static IP — NLB provides static IP addresses. Register the ALB as a target in the NLB's target group. On-prem connects to the fixed NLB IP over DX. The ALB can still scale and change IPs behind the NLB. This is the canonical "static IP for ALB" pattern.
A company needs to query 5 years of archived logs stored in S3 Glacier. Queries are rare and results can take up to 12 hours. They want minimum cost, no ETL pipeline.
S3 Glacier Select + Athena on restored objects — For ad-hoc rare queries: initiate a Glacier Bulk retrieval (cheapest, 5–12 hr), then query with Athena directly on S3 (serverless, pay-per-query). Alternatively, use S3 Glacier Instant Retrieval tier if millisecond access is needed occasionally — higher cost but avoids restore wait.
A company runs workloads across AWS and Azure. They want unified network connectivity between both clouds with consistent private IP routing, without going over the public internet.
AWS Direct Connect + Azure ExpressRoute meeting at a colocation facility — Both clouds connect to a common colocation provider (e.g., Equinix). Traffic stays on private backbone. Alternatively, use a Transit VPC model with SD-WAN appliances on EC2 to bridge both clouds over site-to-site VPN — lower cost but uses internet. DX + ExpressRoute = enterprise-grade.
A web app is deployed in us-east-1. The team wants to add a DR site in eu-west-1. RDS data must be replicated. They can tolerate up to 1 hour of data loss (RPO=1hr) and 4 hours to recover (RTO=4hr).
Pilot Light DR with RDS cross-region read replica — Pilot light keeps a minimal footprint in EU (read replica, no app servers running). On disaster, promote the read replica to primary, launch app tier from AMIs via CloudFormation. Route 53 failover routing switches DNS. RPO = replication lag (well under 1hr). RTO = instance launch time (~mins to 1hr).
An organization enforces that all new S3 buckets must have server-side encryption enabled. Any non-compliant bucket must be auto-remediated within minutes — not just flagged.
AWS Config Rule (s3-bucket-server-side-encryption-enabled) + SSM Automation remediation — Config detects non-compliance, triggers SSM Automation document to apply SSE-S3 or SSE-KMS. Set Config remediation to Auto with concurrency controls. SCP alone can't enforce this retroactively. Config + SSM is the auto-remediation pattern for the exam.
A retail company's checkout service has a dependency on a third-party payment API that occasionally times out. These timeouts cause checkout failures and lost sales. How do you improve resiliency without changing the payment provider?
Circuit breaker pattern via Step Functions + SQS retry queue — Wrap the payment API call in a Step Functions state machine with retry/catch. On repeated failures, route to an SQS queue for async retry. Use exponential backoff with jitter. Alternatively, cache idempotency tokens in ElastiCache and return "pending" status to users while retrying in background.
A company wants developers to be able to deploy to Dev and Staging, but only the CI/CD pipeline (CodePipeline) — never a human — should be able to deploy to Production.
SCP denying all except CodePipeline role in Prod OU + IAM permission boundary — Apply SCP on Prod OU: Deny ec2:*, ecs:UpdateService, cloudformation:* etc. unless the principal ARN matches the CodePipeline execution role. Humans can't deploy even with admin IAM policies. This is the "humans can't deploy to prod" governance pattern.
A company runs microservices on EKS. The security team requires that each pod should only have access to specific AWS services — not the full EC2 node's IAM role.
IAM Roles for Service Accounts (IRSA) — Associate a Kubernetes service account with an IAM role using OIDC identity provider. Each pod using that service account gets scoped IAM credentials via IRSA — not the node's EC2 instance role. This is pod-level IAM isolation on EKS. Do not rely on node instance profile for fine-grained pod access.
An application uses Cognito User Pools for auth. The app needs to call API Gateway, which in turn calls Lambda. The Lambda must verify the caller's identity and access specific DynamoDB rows only belonging to that user.
Cognito JWT → API Gateway Authorizer → Lambda with Cognito sub claim as partition key — API Gateway validates the Cognito JWT via a Cognito authorizer. Lambda extracts the sub (user ID) from the JWT context. DynamoDB key condition uses sub as the partition key so each user only queries their own rows. This is the full serverless AuthZ pattern.
A legacy monolith writes directly to an RDS MySQL database. A new microservice team wants to start consuming those data changes in real time without modifying the monolith code.
AWS DMS with Change Data Capture (CDC) → Kinesis Data Streams — DMS reads the RDS binary log (CDC mode) and publishes change events to Kinesis without any changes to the monolith. New microservices consume from Kinesis independently. This is the strangler fig + CDC pattern — decouple consumers from legacy writes.

Intelligent-Tiering vs manual lifecycle: IT monitors access patterns automatically — no need to predict when objects become cold. Adds $0.0025/1000 objects monitoring fee. Break-even: worth it for objects >128KB with unpredictable access.

EBS vs S3 vs EFS: Never store archive/static data on EBS ($0.08/GB) when S3 ($0.023/GB) or Glacier ($0.004/GB) works. EFS is most expensive ($0.30/GB) — use only for shared POSIX filesystem needs.
Cost traps: S3 Glacier Flexible Retrieval has restore cost ($0.01/GB) + retrieval time (3-5hr) — use Glacier Instant Retrieval when you need occasional fast access. Objects smaller than 128KB cost more with Intelligent-Tiering monitoring fee than they save. Minimum storage duration: 90 days for Glacier IR, 180 days for Deep Archive.

Cost optimisation decisions
Cloud burst with Spot = massive savings: On-prem handles baseline (already paid for). Burst capacity runs on Spot for 10 days/month. Cost = 10 days × 24hr × Spot price. At r5.2xlarge Spot ($0.15/hr) vs On-Demand ($0.50/hr) = 70% less for burst.

Storage Gateway vs full S3 migration: File Gateway exposes S3 as an NFS/SMB mount on-prem. Apps don't change. Data lives in S3 at $0.023/GB instead of on-prem SAN ($200–500/TB/yr). Payback in months.

When to pick Direct Connect over VPN: DX has consistent bandwidth and lower data transfer rates at high volumes. If transferring >1 TB/month, DX data rate ($0.02/GB) beats internet + VPN at scale.
Cost traps: VPN for bulk data transfer (internet egress + VPN fees add up). NAT Gateway data processing charges for hybrid traffic (use DX or VPC endpoints). Storage Gateway in cached mode fetching cold data repeatedly (switch to stored mode if most data is local).
Quick decision framework — pick the cheapest right service
Need a message queue — cost priority
SQS Standard
$0.40/million requests. SQS FIFO is 10× more expensive ($0.40/100k). Use FIFO only when strict ordering + exactly-once is required. For fan-out, SNS → SQS is cheaper than multiple direct queues.
Run a cron job every 5 minutes
EventBridge Scheduler → Lambda
No EC2 instance needed. Lambda billed only during execution. EventBridge scheduler is free for 14M invocations/month. Replacing a t3.micro cron EC2 ($8/month) = $0/month.
Store 10M small config items, <1KB each
DynamoDB On-Demand
10M items × 1KB = ~10GB storage ($2.50/month). On-Demand avoids over-provisioning. RDS for key-value is overkill. ElastiCache too expensive for infrequently accessed data.
Serve a static website with 10M visits/month
S3 + CloudFront
S3 static hosting + CloudFront: ~$1–5/month total. EC2 with nginx: $20–50/month minimum + management. CloudFront 10M HTTPS requests = $0.085. No servers to patch.
Run ML inference on a schedule (once/hour)
SageMaker Serverless Inference
Billed per invocation + ms of compute. No endpoint running when idle. vs ml.t3.medium endpoint ($0.046/hr × 720hr = $33/month idle). Serverless = ~$0 idle cost.
Log aggregation across 50 accounts
CloudWatch cross-account + S3 + Athena
Central S3 bucket (low cost archival). Athena for ad-hoc queries. Avoid shipping everything to a SIEM — ingestion costs add up. Use CloudWatch Log subscriptions to filter before forwarding.
REST API with <1000 req/day
API Gateway (HTTP API) + Lambda
1000 req/day = 30k/month. Lambda free tier: 1M req. API GW HTTP API: $1/M. Total cost: ~$0/month. EC2 for this workload = minimum $8–15/month with near-zero utilisation.
Persistent relational DB for dev/test environments
RDS (stop at night) or Aurora Serverless v2
Stop RDS instances nights/weekends (saves 65% of compute cost — storage still billed). Or Aurora Serverless v2 scales to 0 ACUs when idle. Dev envs should never run 24/7 on provisioned RDS.
Run containerised batch job once/day for 2hr
AWS Batch + Fargate Spot
Batch provisions compute only during job execution. Fargate Spot = 70% off Fargate on-demand. 2hr/day × 30 days = 60hr/month compute. EKS cluster running 24/7 for same job = 720hr/month billed.
Share one NAT Gateway across multiple VPCs
Centralised NAT via Transit Gateway
NAT GW costs $0.045/hr + $0.045/GB processed. 10 VPCs each with own NAT GW = $324/month minimum. 1 central NAT GW + TGW routing = ~$80/month. Savings: $240+/month.
Long-term audit log storage (7 years, rarely read)
S3 Glacier Deep Archive + Object Lock
$0.00099/GB = $1/TB/month. 10TB audit logs = $10/month. S3 Standard = $230/month for same data. Object Lock Compliance mode ensures immutability. CloudTrail → S3 → Lifecycle to Deep Archive at day 90.
Real-time dashboard on streaming data, low budget
Kinesis Firehose → S3 → Athena + QuickSight SPICE
Firehose buffers and delivers to S3 (no per-message cost like KDS). Athena queries for aggregations. QuickSight SPICE caches results — users query cache not Athena. Saves repeated Athena scan costs. SPICE = $0.25/GB/month.
