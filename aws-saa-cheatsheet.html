Design Principles to Know

  • Design for failure — assume everything fails; build resilience
  • Decouple components — use queues, event buses to reduce dependencies
  • Implement elasticity — scale horizontally, not just vertically
  • Think parallel — use multiple instances, sharding, fan-out
  • Keep dynamic data close to compute; static content at edge
  • Use managed services — reduce operational burden
  • Secure at every layer — defense in depth
  • Automate infrastructure provisioning and recovery

RDS — Key Concepts

  • Multi-AZ — synchronous replication to standby; automatic failover; NOT for read scaling
  • Read Replicas — async replication; up to 5; for read scaling; can be cross-region
  • Automated Backups — point-in-time restore; retained 1–35 days
  • Snapshots — manual; kept until deleted; cross-region copy
  • RDS Proxy — connection pooling; reduces failover time; used with Lambda
  • Encryption at rest — KMS; must be set at creation; snapshot copy to encrypt
  • IAM Auth — for MySQL & PostgreSQL; token-based auth
CRITICAL DISTINCTIONMulti-AZ = High Availability (failover). Read Replicas = Scalability (performance). Multi-AZ standby is NOT accessible for reads.

DynamoDB — Key Concepts

  • Partition Key — primary key; uniquely identifies item; must be unique
  • Sort Key — optional; allows multiple items with same partition key
  • GSI — Global Secondary Index; different partition+sort key; separate RCU/WCU
  • LSI — Local Secondary Index; same partition, different sort key; created at table creation only
  • DynamoDB Streams — ordered log of item changes; triggers Lambda
  • Global Tables — multi-region active-active replication
  • DAX — in-memory cache for DynamoDB; microsecond reads; no app code change
  • On-Demand vs Provisioned — on-demand = pay per request; provisioned = set RCU/WCU (use Auto Scaling)
  • TTL — automatically expire items; no extra charge

Auto Scaling Group (ASG)

  • Launch Template — defines instance config (AMI, type, SG, user data)
  • Min/Max/Desired — floor, ceiling, target capacity
  • Scaling Policies:
    • Target Tracking — keep metric at target (e.g. CPU 50%) — simplest
    • Step Scaling — different scale amounts based on alarm breach size
    • Simple Scaling — wait for cooldown; then evaluate again
    • Scheduled — time-based; known traffic patterns
    • Predictive — ML forecast; scale ahead of demand
  • Warm Pools — pre-initialized instances; reduce scale-out latency
  • Instance Refresh — rolling AMI/launch template updates
  • Lifecycle Hooks — pause before launch or terminate; run scripts
  • Default Termination Policy — oldest launch config; AZ balance

SQS Key Concepts

  • Visibility Timeout — time message is hidden after receive; default 30s, max 12hr
  • Dead Letter Queue (DLQ) — failed messages after maxReceiveCount; investigate failures
  • Long Polling — wait up to 20s for messages; reduces empty responses + cost
  • Message Timer — delay individual message delivery (0–15 min)
  • Delay Queue — delay all messages in queue (0–15 min)
  • Message Size — max 256 KB; use S3 + SQS Extended Client for larger
  • FIFO Deduplication — ContentBasedDeduplication or explicit deduplication ID (5-min window)

Common Traps & Gotchas

  • Multi-AZ ≠ Read Replicas — Multi-AZ is HA; Read Replicas scale reads. Standby NOT readable.
  • S3 is NOT block/file storage — cannot mount S3 like EBS/EFS
  • EBS is AZ-locked — cannot attach to instance in different AZ
  • Security Groups are stateful — return traffic auto allowed. NACLs are stateless — need explicit inbound AND outbound rules
  • VPC Peering is non-transitive — A↔B, B↔C does NOT mean A↔C. Use Transit Gateway.
  • DynamoDB RCU math — 1 RCU = 1 strongly consistent read/s of 4KB (or 2 eventually consistent)
  • SQS Standard ≠ ordered — use FIFO for ordering; standard = at-least-once delivery
  • NAT Gateway is per-AZ — create one per AZ for HA; route each AZ's private subnet to local NAT GW
  • CloudFront vs Global Accelerator — CloudFront caches (CDN); GA routes to nearest endpoint (no cache)
  • Cognito User Pools = authentication; Cognito Identity Pools = AWS credentials (authorization)
  • ElastiCache is NOT persistent by default — Redis persistence requires config; not for primary data
  • CloudTrail is enabled by default (90 days event history); create Trail for longer retention to S3

Master Reference — Services to Know Cold

  • EC2 Placement Groups: Cluster (same rack, low lat) / Spread (diff rack) / Partition (Hadoop/Kafka)
  • Aurora Global DB = 1 primary region + up to 5 secondary; 1s replication; sub-1min failover
  • S3 Lifecycle transitions: cannot move Standard → Standard-IA in <30 days
  • KMS CMK = customer managed key; AWS-managed key = free, less control
  • IAM Permission Boundary = max permissions cap; doesn't grant by itself
  • PrivateLink = expose service to other VPC via NLB + Interface Endpoint
  • SSM Session Manager = shell to EC2 without SSH/bastion/port 22
  • Route 53 Geolocation = user country/continent; must have default record
  • CloudFront Origin Access Control (OAC) replaces OAI; restricts S3 to CloudFront
  • EBS io2 Block Express = highest performance; 256K IOPS; multi-attach
  • DynamoDB GSI = new partition+sort key; provisioned capacity separate; query any attribute
  • Lambda@Edge = run Lambda at CloudFront edge for request/response manipulation
  • WAF ACL = attach to ALB, API GW, CloudFront, AppSync; NOT to NLB
  • NLB = static IP per AZ; supports AWS PrivateLink; handles millions req/sec
  • EFS = shared file system; Linux only (NFS); multi-AZ; scales automatically
  • Redshift Spectrum = query S3 data directly without loading into Redshift
  • Macie = PII discovery in S3 only (not EBS/RDS/EFS)
  • Config Aggregator = aggregate Config data across accounts/regions
  • SQS Visibility Timeout default 30s — if consumer takes longer, message reappears
  • Kinesis Shard limit = 1 MB/s write, 2 MB/s read per shard; scale by splitting shards
